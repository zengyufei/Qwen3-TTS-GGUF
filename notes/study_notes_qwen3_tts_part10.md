# Qwen3-TTS 深度笔记 (Part 10)：工匠到底看到了什么？

这是我们出发前的最后一块拼图。你的问题切中了 **Model Dependency (模型依赖关系)** 的核心。

> **Q: 那后面的工匠，是要收这个 code0，还是 code0 加上第21个embedding 叠加起来的和？**

**答案：工匠收的是 [ Code 0 ] + [ 大师的脑波 ]。**

---

## 1. 工匠的特殊待遇

Code Predictor（工匠）虽然是副模型，但它的输入很有意思。在源码中，它的输入是这样的（`inputs_embeds`）：

`Cat( [大师的 Hidden State], [Code 0 的 Embedding] )`

这意味着：
1.  **它不需要再看一遍文本**：因为它拿到了大师的 **Hidden State (隐藏层状态)**。
    *   Hidden State 是大师在处理完上一帧输入（包含文本和音频历史）后，大脑里的“念头”。
    *   这个“念头”里已经融合了文本的含义（比如“要发‘天’的音”）和音频的韵律。
2.  **它只需要看当下的骨架**：即刚刚生成的 **Code 0**。

---

## 2. 完整的接力流程 (The Relay)

让我们分解一下 **第 21 步 (生成“天”的音频)** 的微观操作：

### Step A: 准备阶段
*   我们手里有上一步生成的 `Code0_20`。
*   大师说：“把 `Code0_20` (及它的细节) 和 `Text_20` ('今') 加起来给我。”
*   **大师吃进去** -> **吐出**：
    1.  **Code0_21** (新的骨架，对应 '天')。
    2.  **Hidden_State_20** (大师此时的脑波，包含了 '今' 完成的信息，准备接 '天')。

### Step B: 工匠阶段 (Code Predictor)
*   **输入**：
    1.  **Code0_21** (刚出炉的骨架)。
    2.  **Hidden_State_20** (大师的脑波)。
*   **动作**：
    *   工匠：“大师脑子里想着‘天’，而且给了我这个高度的骨架，那我把剩下的肉填满。”
*   **输出**：`Code1_21` ... `Code15_21`。

### Step C: 闭环
*   现在我们有了完整的第 21 帧音频 (`Code0..15_21`)。
*   把它加起来，再加上第 21 个字 `Text_21` ('天')。
*   **准备喂给大师，开始 Step 22。**

---

## 3. 这里的“和”体现在哪里？

你问的 **“叠加起来的和”**，其实是在 **下一轮循环的开头** (Step C) 才发生的。

*   **Code Predictor 内部**：**不叠加文本**。它只看骨架和脑波。
*   **Talker 输入端**：**必须叠加文本**。这是为了产生下一个骨架。

所以，导出 **Code Predictor (ONNX)** 的时候，我们需要两个输入口：
1.  `talker_hidden` (fp32, [1, 2048])
2.  `code0_embed` (fp32, [1, 2048]) (或者是 index)

它不需要 `text_embed` 作为输入。

这个逻辑理顺了吗？大师负责把文本和历史融合成“脑波”，工匠只需要读懂脑波和骨架就能干活了。
